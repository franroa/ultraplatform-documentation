# The public showcase & publishing model

The public instance of Ultraplatform (the portfolio) demonstrates a stronger claim than a
demo: **the showcase is a fake platform, not fake code.**

- The engine has no fixture/test path. The showcase deployment sets `platformDir` to a
  directory of fictional-but-real files and runs the same pipeline as production.
- The fake platform is deep enough to be worth mapping: 30+ repos of Terraform/YAML/markdown,
  four tenants across four regions, a staged drift feed — and a **GitOps layer** (`up-kubernetes/gitops/
  platform-services/`, sync-waved Argo CD Applications placed by cluster class: `base/`
  everywhere — policy, secrets, autoscaling, scanning, cost allocation — `hub/` once,
  `sandbox/` never in live). The map's AKS view renders exactly that landscape.
- The claims are load-bearing, not decorative. The **tenant publish path** is fully wired
  (one shared gateway, per-tenant `HTTPRoute`s, oauth2-proxy sidecars, a `ReferenceGrant`
  as the only cross-namespace exception, per-tenant rate limits); its resilience story ships
  as code (a `PodDisruptionBudget` plus a scheduled Litmus pod-kill with a continuous probe);
  and the map's **failure-mode lens** (`#v=failure-modes`) renders what each outage scenario
  actually takes down. The staged drift feed even includes an SSO-bypass route, so detection
  demonstrably closes the loop.
- The static site (GitHub Pages, no backend) falls back to a snapshot that is itself
  **generated by running the engine** over that fake platform — never hand-written.
- Everything publish-bound passes a **leak gate**: real names live only in a local
  namemap, and publishing refuses if any of them survives in the artifact set.

The development model is one private source repository; the public repositories
(portfolio, this documentation) are generated artifacts, overwritten on every publish.
The same publish step can also rebuild a private live deployment — one command updates
every consumer.
